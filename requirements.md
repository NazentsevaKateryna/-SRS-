Ось приклад вмісту файлу `requirements.md` для проекту **TaskMaster**:

```markdown
# SRS: TaskMaster

## 1. Мета та сфера застосування
**TaskMaster** – це веб-додаток для управління завданнями, орієнтований на підвищення продуктивності команд. Метою системи є надання користувачам інструменту для створення, моніторингу та управління завданнями в режимі реального часу, а також для ефективної співпраці в межах команди. Додаток повинен забезпечувати простоту використання та підтримку великих команд.

## 2. Функціональні вимоги
### 2.1. Управління завданнями
- Користувачі повинні мати можливість створювати нові завдання з такими полями:
  - Назва завдання
  - Опис завдання
  - Дата виконання
  - Пріоритет (низький, середній, високий)
  - Відповідальна особа
- Користувачі можуть редагувати або видаляти існуючі завдання.
- Користувачі можуть призначати завдання іншим членам команди та змінювати статус виконання (нове, у процесі, виконано).

### 2.2. Командна співпраця
- Можливість створювати та управляти командами (запрошення нових учасників через email).
- Кожна команда повинна мати окремий робочий простір для завдань.
- Реалізувати чат для обговорення завдань у команді.

### 2.3. Повідомлення
- Користувачі повинні отримувати сповіщення про нові або змінені завдання.
- Налаштування сповіщень повинні бути доступні через профіль користувача (вибір email або push-повідомлень).

### 2.4. Дашборд
- Інтерфейс дашборду повинен відображати:
  - Статус завдань у реальному часі.
  - Прогрес виконання завдань команди.
  - Календар із запланованими датами виконання завдань.

## 3. Нефункціональні вимоги
### 3.1. Продуктивність
- Система повинна обробляти до 1000 активних користувачів одночасно без помітного зниження продуктивності.
- Час відгуку на основні операції (створення, редагування завдань) повинен бути не більше 1 секунди.

### 3.2. Безпека
- Користувацькі дані повинні передаватися через захищені канали з використанням HTTPS.
- Аутентифікація користувачів повинна використовувати OAuth 2.0 з можливістю двофакторної автентифікації.

### 3.3. Масштабованість
- Система повинна бути готовою до масштабування для підтримки зростання кількості користувачів до 10 000 активних сесій.

### 3.4. Доступність
- Веб-додаток повинен працювати на всіх основних браузерах (Chrome, Firefox, Safari, Edge).
- Доступність повинна відповідати стандартам WCAG 2.1.

 ### 3.5. Надійність
- Система повинна мати доступність на рівні 99.95% протягом року.
- Усі критичні зміни повинні бути тестовані в середовищі розробки до виходу в продуктивну версію.
```
